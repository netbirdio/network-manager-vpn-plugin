# Installation UX improvements

This note captures the install UX direction beyond the release packages and manual tarball.
A custom package repository/PPA is intentionally deferred.

## Current install paths

GitHub releases should provide:

- `.deb` packages for Debian/Ubuntu-style systems
- `.rpm` packages for Fedora/RHEL-style systems
- generic `tar.gz` archives with `install.sh` and `uninstall.sh`
- SHA-256 checksums generated by GoReleaser

The native packages should remain the preferred path because they can own files cleanly, declare the NetBird runtime and NetworkManager dependencies, and remove installed files safely.

The tarball is an escape hatch for unsupported distributions. It should stay auditable and explicit:

```bash
tar xf nm-netbird-service_linux_amd64.tar.gz
cd nm-netbird-service_linux_amd64
sudo ./install.sh
```

Do not add curl-to-shell as the primary install UX.

## Future doctor helper

A `doctor` command would reduce support friction after install.

Recommended command shape:

```bash
nm-netbird doctor
```

A fallback on the service binary is acceptable if we do not want another executable yet:

```bash
nm-netbird-service --check
```

Prefer a separate user-facing helper long term so the D-Bus service stays focused on NetworkManager integration.

### Behavior

`doctor` should be read-only by default. It should not create connections, mutate NetworkManager config, start login flows, or restart services unless the user passes an explicit repair flag in a later version.

Checks to include:

- NetworkManager is installed and running.
- `nmcli` is available.
- NetBird daemon/runtime is installed.
- NetBird daemon is running, or the configured init system can start it.
- The daemon gRPC socket is reachable at the configured address.
- `nm-netbird-service` is installed at the path referenced by NetworkManager metadata.
- NetworkManager VPN metadata exists and registers `vpn-type netbird`.
- D-Bus policy exists for `org.freedesktop.NetworkManager.netbird`.
- NetBird-owned interfaces such as `wt*` are marked unmanaged by NetworkManager.
- NetworkManager can discover the NetBird VPN plugin after reload/restart.
- At least one NetBird auth path is usable: existing daemon login, setup-key profile data, or SSO-capable interactive activation.

### Output style

Use concise, actionable output:

```text
✓ NetworkManager is running
✓ NetBird daemon socket is reachable: unix:///var/run/netbird.sock
✓ VPN type "netbird" is registered
! NetBird daemon is not logged in

Run:
  netbird login
then:
  nmcli connection up NetBird
```

Recommended exit codes:

- `0`: usable install
- `1`: install is incomplete or activation would fail
- `2`: command usage/configuration error

## Future setup helper

A setup helper should make the first NetworkManager profile easy without doing surprising work at package install time.

Recommended command shape:

```bash
nm-netbird setup
```

Useful options:

```bash
nm-netbird setup --name NetBird
nm-netbird setup --existing-login
nm-netbird setup --setup-key
nm-netbird setup --sso --hint alice@example.com
nm-netbird setup --profile prod --username alice@example.com
nm-netbird setup --activate
nm-netbird setup --dry-run
```

### Behavior

The helper should:

- create or update one NetworkManager VPN connection explicitly requested by the user
- default the connection name to `NetBird`
- use `vpn-type netbird`
- preserve existing unknown `vpn.data` and `vpn.secrets` keys when updating
- store setup keys only in `vpn.secrets`, never `vpn.data`
- support `--dry-run` and print the `nmcli` operations it would perform
- avoid activation unless `--activate` is passed
- print the next command to run after setup

It should not run automatically during package installation. Package install should only place files and reload discovery state.

### First version scope

A pragmatic first version can be a small CLI wrapper around `nmcli`:

1. Check that the plugin is installed and discoverable.
2. Create the connection if it does not exist.
3. Apply selected `vpn.data` and `vpn.secrets` keys.
4. Print `nmcli connection up <name>` or run it when `--activate` is provided.

Avoid building a broader NetBird account manager. The NetBird daemon remains the owner of authentication and networking state.
